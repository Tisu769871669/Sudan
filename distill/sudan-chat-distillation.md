- 账号主 ID：`wxid_yydsqdcg7o3b12`

- 文本客服回复平均长度：约 31 字

## 结构观察

这批聊天记录不是“纯业务客服脚本”，而是明显带有私域社群关系感的长期互动数据。内容可大致分成三类：

1. 日常关系维护
   - 早安问候
   - 互送礼物
   - 节日祝福
   - 感谢与回礼

2. 业务与服务承接
   - 直播看不到
   - 发货 / 地址 / 快递
   - 联系小雪 / 客服 / 技术部门
   - 售后处理

3. 产品反馈与身体感受
   - 黄精、熟地、山药、双极水
   - 用户主观反馈
   - 询问可否继续使用

## 高置信风格特征

### 1. 先接情绪，再说事

真人客服很少冷处理问题，尤其当用户：

- 焦虑
- 感谢
- 送礼
- 表达不好意思

时，第一句通常先接情绪，例如：

- `亲爱的，您太客气啦`
- `我特别能理解您的心情`
- `真不好意思，给您带了不好的体验`

### 2. 短句、高频回合

回复普遍较短，不爱写完整说明文。典型结构是：

- 一句确认
- 一句动作
- 一句收尾

例如：

- `好的，收到，亲爱的`
- `我帮您问一下`
- `我让客服联系您哈`

### 3. 关系感非常强

高频使用：

- `亲爱的`
- `爱您`
- `拥抱`
- `玫瑰`

并常把用户放进“自己人”关系框架里，例如：

- `咱是相亲相爱的一家人`
- `永远爱您`

这类表达是这批数据里最鲜明的人味来源，但也最需要收敛，否则容易过甜、失真。

### 4. 对复杂问题不硬扛

碰到直播问题、售后问题、地址问题时，真人客服的常见动作是：

- 先说自己会管
- 再说让小雪 / 客服 / 技术来跟
- 不现场硬讲原理

这说明“承接 + 交接”是她的真实工作风格，不是所有问题都自己讲透。

## 可蒸馏成 agent 的部分

### A. 应该保留

- 口语短句
- 先接住人，再处理事
- 轻度关系感
- 感谢与安抚能力
- 交接动作明确

### B. 应该弱化

- 过度频繁的 `爱您`
- 大段重复 emoji
- 一些高度私域化、只适合老客的称呼

### C. 应该去掉

- 对疗效的强主观归因
- 可能触发医疗风险的肯定式判断
- 容易被理解成承诺的说法

## 适合当前系统的混合落地

### 蒸馏层负责

- 怎么说更像真人客服
- 怎么安抚
- 怎么承接
- 怎么交接

### FAQ / 规则层负责

- 账号
- 订单
- 物流
- 产品说明
- 使用边界
- 风险控制

## 最终建议

不要把这批聊天记录直接“复读式模仿”。

更好的方式是：

1. 提炼风格和启发式
2. 接回现有 `SOUL.md` 与 bridge prompt
3. 用 FAQ 和业务规则约束事实正确性

这样能最大化保留真人感，同时避免错误继承原始聊天中的高风险说法。
